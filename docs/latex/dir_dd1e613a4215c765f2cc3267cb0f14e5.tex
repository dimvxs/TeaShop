{\bfseries{Light ECMAScript (Java\+Script) \doxylink{class_value}{Value} Notation}} Levn is a library which allows you to parse a string into a Java\+Script value based on an expected type. It is meant for short amounts of human entered data (eg. config files, command line arguments).

Levn aims to concisely describe Java\+Script values in text, and allow for the extraction and validation of those values. Levn uses \href{https://github.com/gkz/type-check}{\texttt{type-\/check}} for its type format, and to validate the results. MIT license. Version 0.\+4.\+1.

{\bfseries{How is this different than JSON?}} levn is meant to be written by humans only, is (due to the previous point) much more concise, can be validated against supplied types, has regex and date literals, and can easily be extended with custom types. On the other hand, it is probably slower and thus less efficient at transporting large amounts of data, which is fine since this is not its purpose. \begin{DoxyVerb}npm install levn
\end{DoxyVerb}

\hypertarget{dir_dd1e613a4215c765f2cc3267cb0f14e5_autotoc_md1965}{}\doxysubsection{\texorpdfstring{Levn Format}{Levn Format}}\label{dir_dd1e613a4215c765f2cc3267cb0f14e5_autotoc_md1965}
Levn can use the type information you provide to choose the appropriate value to produce from the input. For the same input, it will choose a different output value depending on the type provided. For example, {\ttfamily parse(\textquotesingle{}Number\textquotesingle{}, \textquotesingle{}2\textquotesingle{})} will produce the number {\ttfamily 2}, but {\ttfamily parse(\textquotesingle{}String\textquotesingle{}, \textquotesingle{}2\textquotesingle{})} will produce the string {\ttfamily "{}2"{}}.

If you do not provide type information, and simply use {\ttfamily \texorpdfstring{$\ast$}{*}}, levn will parse the input according the unambiguous "{}explicit"{} mode, which we will now detail -\/ you can also set the {\ttfamily explicit} option to true manually in the options.


\begin{DoxyItemize}
\item {\ttfamily "{}string"{}}, {\ttfamily \textquotesingle{}string\textquotesingle{}} are parsed as a String, eg. {\ttfamily "{}a msg"{}} is {\ttfamily "{}a msg"{}}
\item {\ttfamily \#date\#} is parsed as a Date, eg. {\ttfamily \#2011-\/11-\/11\#} is {\ttfamily new Date(\textquotesingle{}2011-\/11-\/11\textquotesingle{})}
\item {\ttfamily /regexp/flags} is parsed as a Reg\+Exp, eg. {\ttfamily /re/gi} is {\ttfamily /re/gi}
\item {\ttfamily undefined}, {\ttfamily null}, {\ttfamily NaN}, {\ttfamily true}, and {\ttfamily false} are all their Java\+Script equivalents
\item {\ttfamily \mbox{[}element1, element2, etc\mbox{]}} is an Array, and the casting procedure is recursively applied to each element. Eg. {\ttfamily \mbox{[}1,2,3\mbox{]}} is {\ttfamily \mbox{[}1,2,3\mbox{]}}.
\item {\ttfamily (element1, element2, etc)} is an tuple, and the casting procedure is recursively applied to each element. Eg. {\ttfamily (1, a)} is {\ttfamily (1, a)} (is {\ttfamily \mbox{[}1, \textquotesingle{}a\textquotesingle{}\mbox{]}}).
\item {\ttfamily \{key1\+: val1, key2\+: val2, ...\}} is an Object, and the casting procedure is recursively applied to each property. Eg. {\ttfamily \{a\+: 1, b\+: 2\}} is {\ttfamily \{a\+: 1, b\+: 2\}}.
\item Any test which does not fall under the above, and which does not contain special characters ({\ttfamily \mbox{[}\`{}\`{}\mbox{]}\`{}\`{}(\`{}\`{})\`{}\`{}\{\`{}\`{}\}\`{}\`{}\+:\`{}\`{},}) is a string, eg. {\ttfamily \$12-\/ blah} is {\ttfamily "{}\$12-\/ blah"{}}.
\end{DoxyItemize}

If you do provide type information, you can make your input more concise as the program already has some information about what it expects. Please see the \href{https://github.com/gkz/type-check\#type-format}{\texttt{type format}} section of \href{https://github.com/gkz/type-check}{\texttt{type-\/check}} for more information about how to specify types. There are some rules about what levn can do with the information\+:


\begin{DoxyItemize}
\item If a String is expected, and only a String, all characters of the input (including any special ones) will become part of the output. Eg. {\ttfamily \mbox{[}(\{\})\mbox{]}} is {\ttfamily "{}\mbox{[}(\{\})\mbox{]}"{}}, and {\ttfamily "{}hi"{}} is {\ttfamily \textquotesingle{}"{}hi"{}\textquotesingle{}}.
\item If a Date is expected, the surrounding {\ttfamily \#} can be omitted from date literals. Eg. {\ttfamily 2011-\/11-\/11} is {\ttfamily new Date(\textquotesingle{}2011-\/11-\/11\textquotesingle{})}.
\item If a Reg\+Exp is expected, no flags need to be specified, and the regex is not using any of the special characters,the opening and closing {\ttfamily /} can be omitted -\/ this will have the affect of setting the source of the regex to the input. Eg. {\ttfamily regex} is {\ttfamily /regex/}.
\item If an Array is expected, and it is the root node (at the top level), the opening {\ttfamily \mbox{[}} and closing {\ttfamily \mbox{]}} can be omitted. Eg. {\ttfamily 1,2,3} is {\ttfamily \mbox{[}1,2,3\mbox{]}}.
\item If a tuple is expected, and it is the root node (at the top level), the opening {\ttfamily (} and closing {\ttfamily )} can be omitted. Eg. {\ttfamily 1, a} is {\ttfamily (1, a)} (is {\ttfamily \mbox{[}1, \textquotesingle{}a\textquotesingle{}\mbox{]}}).
\item If an Object is expected, and it is the root node (at the top level), the opening {\ttfamily \{} and closing {\ttfamily \}} can be omitted. Eg {\ttfamily a\+: 1, b\+: 2} is {\ttfamily \{a\+: 1, b\+: 2\}}.
\end{DoxyItemize}

If you list multiple types (eg. {\ttfamily Number \texorpdfstring{$\vert$}{|} String}), it will first attempt to cast to the first type and then validate -\/ if the validation fails it will move on to the next type and so forth, left to right. You must be careful as some types will succeed with any input, such as String. Thus put String at the end of your list. In non-\/explicit mode, Date and Reg\+Exp will succeed with a large variety of input -\/ also be careful with these and list them near the end if not last in your list.

{\ttfamily custom\+Types} is an object whose keys are the name of the types, and whose values are an object with three properties, {\ttfamily type\+Of}, {\ttfamily validate}, and {\ttfamily cast}. For more information about {\ttfamily type\+Of} and {\ttfamily validate}, please see the \href{https://github.com/gkz/type-check\#custom-types}{\texttt{custom types}} section of type-\/check.

{\ttfamily cast} is a function which receives three arguments, the value under question, options, and the types\+Cast function. In {\ttfamily cast}, attempt to cast the value into the specified type. If you are successful, return an object in the format {\ttfamily \{type\+: \textquotesingle{}Just\textquotesingle{}, value\+: CAST-\/\+VALUE\}}, if you know it won\textquotesingle{}t work, return {\ttfamily \{type\+: \textquotesingle{}Nothing\textquotesingle{}\}}. You can use the {\ttfamily types\+Cast} function to cast any child values. Remember to pass {\ttfamily options} to it. In your function you can also check for {\ttfamily options.\+explicit} and act accordingly.\hypertarget{dir_dd1e613a4215c765f2cc3267cb0f14e5_autotoc_md1969}{}\doxysubsection{\texorpdfstring{Technical About}{Technical About}}\label{dir_dd1e613a4215c765f2cc3267cb0f14e5_autotoc_md1969}
{\ttfamily levn} is written in \href{http://livescript.net/}{\texttt{Live\+Script}} -\/ a language that compiles to Java\+Script. It uses \href{https://github.com/gkz/type-check}{\texttt{type-\/check}} to both parse types and validate values. It also uses the \href{http://preludels.com/}{\texttt{prelude.\+ls}} library.
