ESLint is a tool for identifying and reporting on patterns found in ECMAScript/\+Java\+Script code. In many ways, it is similar to JSLint and JSHint with a few exceptions\+:


\begin{DoxyItemize}
\item ESLint uses \href{https://github.com/eslint/js/tree/main/packages/espree}{\texttt{Espree}} for Java\+Script parsing.
\item ESLint uses an AST to evaluate patterns in code.
\item ESLint is completely pluggable, every single rule is a plugin and you can add more at runtime.

The names {\ttfamily "{}prefer-\/const"{}} and {\ttfamily "{}no-\/constant-\/binary-\/expression"{}} are the names of \href{https://eslint.org/docs/rules}{\texttt{rules}} in ESLint. The first value is the error level of the rule and can be one of these values\+:


\begin{DoxyItemize}
\item {\ttfamily "{}off"{}} or {\ttfamily 0} -\/ turn the rule off
\item {\ttfamily "{}warn"{}} or {\ttfamily 1} -\/ turn the rule on as a warning (doesn\textquotesingle{}t affect exit code)
\item {\ttfamily "{}error"{}} or {\ttfamily 2} -\/ turn the rule on as an error (exit code will be 1)

\hypertarget{dir_8cbfb7c497e480124fd777f30e57b1f2_autotoc_md1303}{}\doxysubsubsubsection{\texorpdfstring{Does ESLint support JSX?}{Does ESLint support JSX?}}\label{dir_8cbfb7c497e480124fd777f30e57b1f2_autotoc_md1303}
Yes, ESLint natively supports parsing JSX syntax (this must be enabled in \href{https://eslint.org/docs/latest/use/configure}{\texttt{configuration}}). Please note that supporting JSX syntax {\itshape is not} the same as supporting React. React applies specific semantics to JSX syntax that ESLint doesn\textquotesingle{}t recognize. We recommend using \href{https://www.npmjs.com/package/eslint-plugin-react}{\texttt{eslint-\/plugin-\/react}} if you are using React and want React semantics.\hypertarget{dir_8cbfb7c497e480124fd777f30e57b1f2_autotoc_md1304}{}\doxysubsubsubsection{\texorpdfstring{Does Prettier replace ESLint?}{Does Prettier replace ESLint?}}\label{dir_8cbfb7c497e480124fd777f30e57b1f2_autotoc_md1304}
No, ESLint and Prettier have different jobs\+: ESLint is a linter (looking for problematic patterns) and Prettier is a code formatter. Using both tools is common, refer to \href{https://prettier.io/docs/en/install\#eslint-and-other-linters}{\texttt{Prettier\textquotesingle{}s documentation}} to learn how to configure them to work well with each other.\hypertarget{dir_8cbfb7c497e480124fd777f30e57b1f2_autotoc_md1305}{}\doxysubsubsubsection{\texorpdfstring{What ECMAScript versions does ESLint support?}{What ECMAScript versions does ESLint support?}}\label{dir_8cbfb7c497e480124fd777f30e57b1f2_autotoc_md1305}
ESLint has full support for ECMAScript 3, 5, and every year from 2015 up until the most recent stage 4 specification (the default). You can set your desired ECMAScript syntax and other settings (like global variables) through \href{https://eslint.org/docs/latest/use/configure}{\texttt{configuration}}.\hypertarget{dir_8cbfb7c497e480124fd777f30e57b1f2_autotoc_md1306}{}\doxysubsubsubsection{\texorpdfstring{What about experimental features?}{What about experimental features?}}\label{dir_8cbfb7c497e480124fd777f30e57b1f2_autotoc_md1306}
ESLint\textquotesingle{}s parser only officially supports the latest final ECMAScript standard. We will make changes to core rules in order to avoid crashes on stage 3 ECMAScript syntax proposals (as long as they are implemented using the correct experimental ESTree syntax). We may make changes to core rules to better work with language extensions (such as JSX, Flow, and Type\+Script) on a case-\/by-\/case basis.

In other cases (including if rules need to warn on more or fewer cases due to new syntax, rather than just not crashing), we recommend you use other parsers and/or rule plugins. If you are using Babel, you can use \href{https://www.npmjs.com/package/@babel/eslint-parser}{\texttt{@babel/eslint-\/parser}} and \href{https://www.npmjs.com/package/@babel/eslint-plugin}{\texttt{@babel/eslint-\/plugin}} to use any option available in Babel.

Once a language feature has been adopted into the ECMAScript standard (stage 4 according to the \href{https://tc39.github.io/process-document/}{\texttt{TC39 process}}), we will accept issues and pull requests related to the new feature, subject to our \href{https://eslint.org/docs/latest/contribute}{\texttt{contributing guidelines}}. Until then, please use the appropriate parser and plugin(s) for your experimental feature.\hypertarget{dir_8cbfb7c497e480124fd777f30e57b1f2_autotoc_md1307}{}\doxysubsubsubsection{\texorpdfstring{Which Node.\+js versions does ESLint support?}{Which Node.\+js versions does ESLint support?}}\label{dir_8cbfb7c497e480124fd777f30e57b1f2_autotoc_md1307}

Open a \href{https://github.com/eslint/eslint/discussions}{\texttt{discussion}} or stop by our \href{https://eslint.org/chat}{\texttt{Discord server}}.\hypertarget{dir_8cbfb7c497e480124fd777f30e57b1f2_autotoc_md1309}{}\doxysubsubsubsection{\texorpdfstring{Why doesn\textquotesingle{}t ESLint lock dependency versions?}{Why doesn\textquotesingle{}t ESLint lock dependency versions?}}\label{dir_8cbfb7c497e480124fd777f30e57b1f2_autotoc_md1309}
Lock files like {\ttfamily package-\/lock.\+json} are helpful for deployed applications. They ensure that dependencies are consistent between environments and across deployments.

Packages like {\ttfamily eslint} that get published to the npm registry do not include lock files. {\ttfamily npm install eslint} as a user will respect version constraints in ESLint\textquotesingle{}s {\ttfamily package.\+json}. ESLint and its dependencies will be included in the user\textquotesingle{}s lock file if one exists, but ESLint\textquotesingle{}s own lock file would not be used.

We intentionally don\textquotesingle{}t lock dependency versions so that we have the latest compatible dependency versions in development and CI that our users get when installing ESLint in a project.

The Twilio blog has a \href{https://www.twilio.com/blog/lockfiles-nodejs}{\texttt{deeper dive}} to learn more.\hypertarget{dir_8cbfb7c497e480124fd777f30e57b1f2_autotoc_md1310}{}\doxysubsubsection{\texorpdfstring{Releases}{Releases}}\label{dir_8cbfb7c497e480124fd777f30e57b1f2_autotoc_md1310}

Read the full \href{https://github.com/eslint/.github/blob/master/SECURITY.md}{\texttt{security policy}}.\hypertarget{dir_8cbfb7c497e480124fd777f30e57b1f2_autotoc_md1312}{}\doxysubsubsection{\texorpdfstring{Semantic Versioning Policy}{Semantic Versioning Policy}}\label{dir_8cbfb7c497e480124fd777f30e57b1f2_autotoc_md1312}
ESLint follows \href{https://semver.org}{\texttt{semantic versioning}}. However, due to the nature of ESLint as a code quality tool, it\textquotesingle{}s not always clear when a minor or major version bump occurs. To help clarify this for everyone, we\textquotesingle{}ve defined the following semantic versioning policy for ESLint\+:


\begin{DoxyItemize}
\item Patch release (intended to not break your lint build)
\begin{DoxyItemize}
\item A bug fix in a rule that results in ESLint reporting fewer linting errors.
\item A bug fix to the CLI or core (including formatters).
\item Improvements to documentation.
\item Non-\/user-\/facing changes such as refactoring code, adding, deleting, or modifying tests, and increasing test coverage.
\item Re-\/releasing after a failed release (i.\+e., publishing a release that doesn\textquotesingle{}t work for anyone).
\end{DoxyItemize}
\item Minor release (might break your lint build)
\begin{DoxyItemize}
\item A bug fix in a rule that results in ESLint reporting more linting errors.
\item A new rule is created.
\item A new option to an existing rule that does not result in ESLint reporting more linting errors by default.
\item A new addition to an existing rule to support a newly-\/added language feature (within the last 12 months) that will result in ESLint reporting more linting errors by default.
\item An existing rule is deprecated.
\item A new CLI capability is created.
\item New capabilities to the public API are added (new classes, new methods, new arguments to existing methods, etc.).
\item A new formatter is created.
\item {\ttfamily eslint\+:recommended} is updated and will result in strictly fewer linting errors (e.\+g., rule removals).
\end{DoxyItemize}
\item Major release (likely to break your lint build)
\begin{DoxyItemize}
\item {\ttfamily eslint\+:recommended} is updated and may result in new linting errors (e.\+g., rule additions, most rule option updates).
\item A new option to an existing rule that results in ESLint reporting more linting errors by default.
\item An existing formatter is removed.
\item Part of the public API is removed or changed in an incompatible way. The public API includes\+:
\begin{DoxyItemize}
\item Rule schemas
\item Configuration schema
\item Command-\/line options
\item Node.\+js API
\item Rule, formatter, parser, plugin APIs
\end{DoxyItemize}
\end{DoxyItemize}
\end{DoxyItemize}

According to our policy, any minor update may report more linting errors than the previous release (ex\+: from a bug fix).
