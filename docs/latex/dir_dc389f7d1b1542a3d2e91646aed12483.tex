\begin{quote}
Simple key-\/value storage with support for multiple backends \end{quote}


\href{https://github.com/jaredwray/keyv/actions/workflows/tests.yaml}{\texttt{}} \href{https://codecov.io/gh/jaredwray/keyv}{\texttt{}} \href{https://www.npmjs.com/package/keyv}{\texttt{}} \href{https://www.npmjs.com/package/keyv}{\texttt{}}

Keyv provides a consistent interface for key-\/value storage across multiple backends via storage adapters. It supports TTL based expiry, making it suitable as a cache or a persistent key-\/value store.\hypertarget{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1914}{}\doxysubsubsection{\texorpdfstring{Features}{Features}}\label{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1914}
There are a few existing modules similar to Keyv, however Keyv is different because it\+:


\begin{DoxyItemize}
\item Isn\textquotesingle{}t bloated
\item Has a simple Promise based API
\item Suitable as a TTL based cache or persistent key-\/value store
\item Easily embeddable inside another module
\item Works with any storage that implements the \href{https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Map}{\texttt{{\ttfamily Map}}} API
\item Handles all JSON types plus {\ttfamily Buffer}
\item Supports namespaces
\item Wide range of \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}efficient, well tested\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*} storage adapters
\item Connection errors are passed through (db failures won\textquotesingle{}t kill your app)
\item Supports the current active LTS version of Node.\+js or higher
\end{DoxyItemize}\hypertarget{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1915}{}\doxysubsubsection{\texorpdfstring{Usage}{Usage}}\label{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1915}

Keyv uses \href{https://github.com/dominictarr/json-buffer}{\texttt{{\ttfamily json-\/buffer}}} for data serialization to ensure consistency across different backends.

You can optionally provide your own serialization functions to support extra data types or to serialize to something other than JSON.

\end{DoxyCode}

Keyv is designed to be easily embedded into other modules to add cache support. The recommended pattern is to expose a {\ttfamily cache} option in your modules options which is passed through to Keyv. Caching will work in memory by default and users have the option to also install a Keyv storage adapter and pass in a connection string, or any other storage that implements the {\ttfamily Map} API.

You should also set a namespace for your module so you can safely call {\ttfamily .clear()} without clearing unrelated app data.

1) Fork this repository into your Git account 2) Clone the forked repository to your local directory using {\ttfamily git clone} 3) Install any of the above missing dependencies\hypertarget{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1944}{}\doxysubsubsection{\texorpdfstring{Launching the project}{Launching the project}}\label{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1944}
Once the project is installed locally, you are ready to start up its services\+:

1) Ensure that your Docker service is running. 2) From the root directory of your project, run the {\ttfamily yarn} command in the command prompt to install yarn. 3) Run the {\ttfamily yarn bootstrap} command to install any necessary dependencies. 4) Run {\ttfamily yarn test\+:services\+:start} to start up this project\textquotesingle{}s Docker container. The container will launch all services within your workspace.\hypertarget{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1945}{}\doxysubsubsection{\texorpdfstring{Available Commands}{Available Commands}}\label{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1945}
Once the project is running, you can execute a variety of commands. The root workspace and each subpackage contain a {\ttfamily package.\+json} file with a {\ttfamily scripts} field listing all the commands that can be executed from that directory. This project also supports native {\ttfamily yarn}, and {\ttfamily docker} commands.

Here, we\textquotesingle{}ll cover the primary commands that can be executed from the root directory. Unless otherwise noted, these commands can also be executed from a subpackage. If executed from a subpackage, they will only affect that subpackage, rather than the entire workspace.\hypertarget{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1946}{}\doxysubsubsubsection{\texorpdfstring{{\ttfamily yarn}}{{\ttfamily yarn}}}\label{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1946}
The {\ttfamily yarn} command installs yarn in the workspace.\hypertarget{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1947}{}\doxysubsubsubsection{\texorpdfstring{{\ttfamily yarn bootstrap}}{{\ttfamily yarn bootstrap}}}\label{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1947}
The {\ttfamily yarn bootstrap} command installs all dependencies in the workspace.\hypertarget{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1948}{}\doxysubsubsubsection{\texorpdfstring{{\ttfamily yarn test\+:services\+:start}}{{\ttfamily yarn test\+:services\+:start}}}\label{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1948}
The {\ttfamily yarn test\+:services\+:start} command starts up the project\textquotesingle{}s Docker container, launching all services in the workspace. This command must be executed from the root directory.\hypertarget{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1949}{}\doxysubsubsubsection{\texorpdfstring{{\ttfamily yarn test\+:services\+:stop}}{{\ttfamily yarn test\+:services\+:stop}}}\label{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1949}
The {\ttfamily yarn test\+:services\+:stop} command brings down the project\textquotesingle{}s Docker container, halting all services. This command must be executed from the root directory.\hypertarget{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1950}{}\doxysubsubsubsection{\texorpdfstring{{\ttfamily yarn test}}{{\ttfamily yarn test}}}\label{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1950}
The {\ttfamily yarn test} command runs all tests in the workspace.\hypertarget{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1951}{}\doxysubsubsubsection{\texorpdfstring{{\ttfamily yarn clean}}{{\ttfamily yarn clean}}}\label{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1951}
The {\ttfamily yarn clean} command removes yarn and all dependencies installed by yarn. After executing this command, you must repeat the steps in {\itshape Setting up your workspace} to rebuild your workspace.\hypertarget{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1952}{}\doxysubsubsection{\texorpdfstring{Contributing Changes}{Contributing Changes}}\label{dir_dc389f7d1b1542a3d2e91646aed12483_autotoc_md1952}
Now that you\textquotesingle{}ve set up your workspace, you\textquotesingle{}re ready to contribute changes to the {\ttfamily keyv} repository.

1) Make any changes that you would like to contribute in your local workspace. 2) After making these changes, ensure that the project\textquotesingle{}s tests still pass by executing the {\ttfamily yarn test} command in the root directory. 3) Commit your changes and push them to your forked repository. 4) Navigate to the original {\ttfamily keyv} repository and go the {\itshape Pull Requests} tab. 5) Click the {\itshape New pull request} button, and open a pull request for the branch in your repository that contains your changes. 6) Once your pull request is created, ensure that all checks have passed and that your branch has no conflicts with the base branch. If there are any issues, resolve these changes in your local repository, and then commit and push them to git. 7) Similarly, respond to any reviewer comments or requests for changes by making edits to your local repository and pushing them to Git. 8) Once the pull request has been reviewed, those with write access to the branch will be able to merge your changes into the {\ttfamily keyv} repository.
